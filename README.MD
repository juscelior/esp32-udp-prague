# ESP32 UDP Prague Client

## Overview

This repository contains an ESP32 UDP client that uses the Prague congestion control algorithm and interoperates with a reference UDP Prague server. The goal is to generate reproducible experiments for L4S/ECN evaluation using an ESP32 microcontroller as the sender and a Linux host as the receiver.

This code and dataset support the MSc dissertation *"Implementation and Evaluation of the L4S Protocol in the Internet of Things"* (Universidade da Beira Interior, 2026). The dissertation reports eight experimental conditions (C1–C8); the mapping between those conditions and the directories in `experiments/` is given below.

The UDP server component is based on a dedicated fork to ensure version stability and compatibility across different environments.

## Server implementation

The UDP Prague server used by this project is available at:

- **Repository**: [juscelior/udp_prague](https://github.com/juscelior/udp_prague)

The fork is used to:
- Preserve a known working version of the UDP Prague server.
- Avoid breaking changes if the original L4S repository evolves.
- Provide a stable reference point for all experiments.

### Local copy in `udp_prague_base/`

This repository also contains a local snapshot of the server sources under the folder `udp_prague_base/`.

- The files in `udp_prague_base/` are a copy of the code from the repository [juscelior/udp_prague](https://github.com/juscelior/udp_prague).
- The repository `juscelior/udp_prague` itself is a fork of [L4STeam/udp_prague](https://github.com/L4STeam/udp_prague).
- This copy is kept only to record exactly which version of the server was used when this work was developed.

```text

  └── ...                      # Other original files kept only as reference

experiments/                   # Experimental dataset (see mapping table below)
  └── tNN-ecn-sender-.../      # One directory per test configuration
      ├── server_N.log         # Receiver log, run N (N = 1..3)
      └── *_runN_client.csv    # Parsed ESP32 client telemetry, run N

scripts/                       # Helper scripts
  ├── configure_and_flash_only.sh   # Sets firmware macros per scenario and flashes
  ├── monitor_esp32_serial.sh       # Resets the board and captures serial output
  └── set_test.sh / set_test.py     # Configures the gateway bottleneck per test

                               # and figure of the dissertation from the raw logs
```

The recommended and maintained path for new ESP32 experiments is the PlatformIO project under `esp32dev/`. The folder `udp_prague_base/` is *not* meant to be modified; it is kept only as a frozen copy of the server code used during this work.

## Experimental dataset (`experiments/`)

### Directory naming convention

```text
tNN-ecn-sender-{ect0|ect1}-{prague|cubic|reno}-{dualpi2|fqcodel}-ecn{0|1|2}
                 │            │                  │                  │
                 │            │                  │                  └─ gateway net.ipv4.tcp_ecn
                 │            │                  └─ AQM qdisc at the emulated bottleneck
                 │            └─ gateway net.ipv4.tcp_congestion_control
                 └─ ECN codepoint marked by the ESP32 sender
```

Two of these tokens describe **gateway TCP settings** (`prague|cubic|reno` and `ecn0|1|2`). The dissertation shows empirically (Section 4.3) that both are **inert for the forwarded UDP flow** — they govern only the gateway's own TCP sockets. The factors that actually shape the results are the client congestion controller (Prague vs. the fixed-window baseline sender), the AQM, and the sender ECN codepoint.

All data-carrying runs use the 512 kbit/s emulated bottleneck (`scripts/set_test.sh`), the `HIGH` firmware scenario (300 s per run), three runs per configuration, and a receiver restart before every run.

### Mapping to the dissertation conditions C1–C8

| Directory (3 runs each) | Client CC | AQM | Sender ECN | Gateway CC / tcp_ecn | Dissertation |
|---|---|---|---|---|---|
| `t04-...-ect1-prague-dualpi2-ecn0` | Prague | DualPI2 | ECT(1) | prague / 0 | **C1** (target, full L4S) |
| `t04-...-ect0-prague-dualpi2-ecn0` | Prague | DualPI2 | Not-ECT | prague / 0 | **C2** |
| `t01-...-ect1-prague-fqcodel-ecn0` | Prague | FQ-CoDel | ECT(1) | prague / 0 | **C3** |
| `t01-...-ect0-prague-fqcodel-ecn0` | Prague | FQ-CoDel | Not-ECT | prague / 0 | **C4** |
| `t16-...-ect1-reno-dualpi2-ecn0` | Baseline | DualPI2 | ECT(1) | reno / 0 | **C5** |
| `t16-...-ect0-reno-dualpi2-ecn0` | Baseline | DualPI2 | Not-ECT | reno / 0 | **C6** |
| `t07-...-ect1-cubic-fqcodel-ecn0` | Baseline | FQ-CoDel | ECT(1) | cubic / 0 | **C7** |
| `t07-...-ect0-cubic-fqcodel-ecn0` | Baseline | FQ-CoDel | Not-ECT | cubic / 0 | **C8** (legacy baseline) |

"Baseline" is the firmware's `CC_MODE_BASELINE`: a fixed window of 10 packets that ignores all ACK feedback. It runs **no congestion control algorithm at all** — the `reno`/`cubic` tokens in those directory names refer to the gateway setting, not to the client.

### Replication runs (t05, t06, t10, t11, t12)

| Directories | Contents | Role |
|---|---|---|
| `t05-*`, `t06-*` (prague gateway, tcp_ecn 1/2) | Prague + DualPI2 + ECT(1) | Replications of C1 |
| `t10-*`, `t11-*`, `t12-*` (cubic gateway, tcp_ecn 0/1/2) | Prague + DualPI2 + ECT(1) | Replications of C1 |

These 30 runs differ from C1 **only in the inert gateway dimensions** and are statistically consistent with it. They serve two purposes in the dissertation: they are the empirical proof that the gateway TCP settings have no effect on the UDP flow (Section 4.3), and they are the additional replication runs that support the C1 results (Section 4.4).

> **⚠ Label caveat:** the `ect0`-labelled directories among t05/t06/t10–t12 actually contain **ECT(1)** traffic. They were collected before a firmware fix to `snd_ecn` (the `ECN_SENDER_ENABLE 0` build still marked ECT(1) at send time; see the dissertation, Chapter 3). This is why their receiver logs show CE marks despite the `ect0` name. They are therefore counted as C1-class replications, **not** as Not-ECT conditions. All C1–C8 directories in the table above post-date the fix and are correctly labelled.

### Empty placeholders

`t02`, `t03`, `t08`, `t09`, `t13`, `t14`, `t15`, `t17`, `t18` contain empty `server_N.log` files and no client telemetry. They are scaffolding from the originally planned full matrix (gateway CC × tcp_ecn × AQM × sender ECN × client CC). After the inertness result of Section 4.3, completing them would only have duplicated C1–C8 under inert relabelings, so they were not run. They are kept so the directory numbering remains continuous and the planned matrix remains visible.

## ESP32 client (PlatformIO)

### Prerequisites

- **Hardware**: ESP32 development board with Wi‑Fi capability.
- **Software**:
  - Visual Studio Code.
  - [PlatformIO IDE extension](https://platformio.org/install/ide?install=vscode).
  - USB driver for the ESP32 so that a serial port such as `/dev/tty.usbserial-*` is available.

### PlatformIO configuration

The `esp32dev/platformio.ini` file should contain a configuration similar to:

```ini
[env:esp32dev]
platform = espressif32
board = esp32dev
framework = arduino

monitor_speed = 115200
```

### Building and uploading

From VS Code with PlatformIO:

1. Open the folder `esp32dev/` as the workspace.
2. Connect the ESP32 board via USB.
3. Use the PlatformIO side bar:
   - **Build** to compile the firmware.
   - **Upload** to flash the firmware to the board.

Alternatively, from a terminal inside `esp32dev/`:

```bash
pio run           # build
pio run -t upload # build and upload
```

For the experiment matrix, `scripts/configure_and_flash_only.sh <test-name>` sets the firmware macros (`ECN_SENDER_ENABLE`, `CC_MODE_BASELINE`, `MAX_WINDOW_ESP32`) automatically from the test name and flashes the board.

### Wi‑Fi and server configuration

In `esp32dev/src/UDPPragueClient.ino`, configure your Wi‑Fi network and the target server:

```cpp
// Wi-Fi credentials
const char* ssid     = "YOUR_SSID";
const char* password = "YOUR_PASSWORD";

// Server configuration
const char* server_ip   = "YOUR_SERVER_IP"; // e.g., Azure public IP
const int   server_port = 5005;             // UDP Prague server port
```

The sketch also exposes several **test scenarios** (baseline, medium, high load, burst, jumbo) that define payload size and duration. You can enable one scenario at a time by defining the corresponding macro near the top of `UDPPragueClient.ino`. The dataset above was collected with the `HIGH` scenario (1383-byte payload, 300 s).

## Capturing ESP32 logs (CSTATS)

The ESP32 client emits compact, structured log lines for analysis. Each acknowledgment from the server produces a line with the following format:

```text
CSTATS;ms_since_start;recv_pkts;inflight;rtt_us;jitter_us;win;burst;rate_Bps
```

These lines can be captured from the serial port and later processed in Python or a Jupyter notebook.

### Using PlatformIO serial monitor

From the `esp32dev/` directory, run:

```bash
pio device monitor
```

To capture and save all output to a file for later analysis:

```bash
pio device monitor | tee ../esp_client.log
```

For experiment runs, `scripts/monitor_esp32_serial.sh <test-name> <run>` resets the board first (so every run starts from the boot banner) and saves the capture with the right name.

The resulting log files are parsed by the notebook `metrics_academic.ipynb`, which reproduces every table, figure and derived number of the dissertation from the raw logs.

## Server-side capture example

On the Linux server running the UDP Prague receiver you can use `tcpdump` to observe incoming traffic:

```bash
sudo tcpdump -i any -vvv -n 'udp and port 5005'
```

> **Important:** restart the receiver before every run. Its internal clock is a 32-bit microsecond counter that wraps after ~35.8 minutes of process lifetime, and its packet counters are cumulative per process — a long-lived receiver corrupts the per-run statistics.

## Testbed used in the dissertation

- ESP32-WROOM-32 (client, Covilhã) → Raspberry Pi 4 gateway → Azure VM East US 2 (receiver).
- Gateway: Linux kernel `6.12.54-l4s` (L4S-patched), `hostapd` Wi-Fi AP, NAT, `sch_dualpi2`.
- Emulated bottleneck: 512 kbit/s `htb` class at the gateway egress with a `u32` filter scoped to the receiver IP, with `dualpi2` or `fq_codel` as the child qdisc (`scripts/set_test.sh`).
